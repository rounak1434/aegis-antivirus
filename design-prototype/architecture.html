Detection pipeline — four layers, one verdict

each file/object flows left → right; early high-confidence hits short-circuit
STAGE A

Signature

Exact, fast, zero false-positives.
  • SHA-256 & MD5 lookup
  • Memory-mapped hash sets
  • Local cache + signed DB updates
  • Known-good allowlist skip
STAGE B

Heuristic

Static rules & suspicious traits.
  • Suspicious import/API usage
  • Injection & persistence markers
  • Obfuscation / packer signatures
  • PowerShell & script abuse
STAGE C

Behavioral

Runtime activity (RTP / sandbox).
  • File-system & registry events
  • Process & network activity
  • Mass-encryption (ransomware) heuristic
  • Sequence correlation
STAGE D

ML scoring

Feature model for the unknown.
  • PE structural features
  • Shannon entropy analysis
  • String & metadata features
  • Calibrated risk score 0–100
Verdict aggregator
Combines layer outputs into a weighted risk score and severity (clean / suspicious / malicious), attaches the matched evidence, and emits a Detection record → UI + quarantine recommendation. Weights and thresholds are configurable per scan mode.

Security model & privilege separation

  • Two-process split. The UI runs unprivileged; only the engine service runs as SYSTEM. A compromised UI can't touch protected resources directly — it can only send authenticated IPC requests.
  • No unsafe by default. Each crate sets #![forbid(unsafe_code)]; OS calls that require it are isolated in aegis-platform behind audited wrappers.
  • Tamper protection. Service binary & signature DB are integrity-checked; the service rejects IPC from unsigned clients and guards its own files/registry keys.
  • Signed updates. Signature/rule updates are delta-downloaded over TLS, verified against a pinned public key, and atomically swapped with rollback on failure.
  • Encrypted quarantine. Isolated files are AES-GCM encrypted at rest with a key in the OS keystore (DPAPI on Windows); originals are neutralised on capture.
  • Local-first & private. All data stays on the device. No cloud lookups unless the user explicitly opts into a community reputation feed.

Build roadmap — 7 phases

each phase ships a usable vertical slice
01DONE-ISH

Architecture & scaffolding this doc

Cargo workspace, crate boundaries, IPC schema, platform trait surface, SQLite schema, and the two-process security model.
Exit: empty crates compile; UI ↔ service IPC round-trips a ping; CI on Windows + Linux.
02

Scanner engine

Recursive walker (hidden/system files), file classifier, double-extension & masquerade detection, packed-executable flag, archive extraction with depth/bomb guards, multithreaded job pool, incremental cache.
Exit: Quick/Full/Deep scans enumerate and hash a real disk with progress events and cancellation.
03

Detection engine

Signature matching (SHA-256/MD5 + rules), heuristic ruleset, PE feature extraction + entropy + string analysis, calibrated risk scoring, and the verdict aggregator. Memory & persistence scanners.
Exit: detects EICAR + curated test corpus; false-positive rate measured on a clean baseline.
04

Real-time protection

File create/modify and process-execution hooks (minifilter / ETW), download monitoring, controlled-folder ransomware rules, behavioral correlation feeding stage C.
Exit: on-access blocking with acceptable overhead; ransomware canary test triggers a block.
05

GUI

The Tauri + React surfaces in this prototype: dashboard, live scan, threats & report, quarantine, real-time, settings/updates, tray + mini widget.
Exit: every engine capability is reachable and observable from the UI; tray reflects live state.
06

Testing & hardening

Unit + integration tests per crate, malware-corpus regression suite (in an isolated VM), fuzzing of archive/PE parsers, performance benchmarks, and a tamper-protection audit.
Exit: green CI, fuzz targets stable, detection & FP metrics tracked over time.
07

Packaging & release

Code-signed MSI/MSIX installer, service registration, signed signature feed + auto-update channel, reproducible builds, and release docs.
Exit: signed installer deploys the service + UI; updates verify and roll back cleanly.

Known limitations

be honest
  • Kernel-level threats & bootkits need a signed driver — out of scope until a driver-signing path exists.
  • The ML layer is heuristic, not a guarantee — risk scores reduce but don't eliminate false positives/negatives.
  • RAR/7z formats may require bundling/licensing decompression libraries; encrypted archives can't be inspected.
  • Real-time hooks add I/O latency; needs careful exclusion lists to stay usable on busy workstations.

Future improvements

roadmap

  • YARA rule import so researchers can extend detection without recompiling.
